CRUD de Clientes con Vue.js, Vite y Nuxt UI

Documento de requisitos y guia didactica para explicar a estudiantes como funciona un CRUD frontend de clientes: creacion, lectura, actualizacion, eliminacion, busqueda, filtrado, seleccion multiple, paginacion y persistencia local.

Producto Dashboard de clientes
Tipo Frontend solamente
Archivo principal src/pages/customers.vue
Version PRD 1.0
Vue 3 Composition API Nuxt UI para Vue TypeScript Zod localStorage

1. Contexto del producto

La aplicacion es un dashboard construido con una plantilla oficial de Nuxt UI para Vue. Dentro de esa plantilla se implemento una pantalla llamada Clientes, cuyo objetivo es permitir gestionar una lista de clientes sin usar backend ni base de datos externa.

El CRUD es ideal para clase porque concentra conceptos esenciales de desarrollo frontend moderno: estado reactivo, formularios, validacion, tablas, componentes reutilizables, eventos, renderizado condicional, persistencia local y experiencia de usuario.

Idea central para estudiantes: aunque no exista una API real, el frontend puede simular el ciclo completo de gestion de datos usando estado local y localStorage. Esto permite practicar la logica del CRUD antes de conectarlo a un backend.

2. Objetivos

Objetivos del producto

  • Mostrar una lista de clientes en una tabla clara y navegable.
  • Permitir crear nuevos clientes desde un formulario modal.
  • Permitir editar la informacion de un cliente existente.
  • Permitir eliminar un cliente individual o varios clientes seleccionados.
  • Conservar los datos en el navegador mediante localStorage.
  • Ofrecer busqueda, filtro por estado, paginacion y control de columnas.

Objetivos de aprendizaje

  • Comprender la diferencia entre estado reactivo y datos persistidos.
  • Aplicar ref, reactive, computed y watch.
  • Crear formularios validados con Zod y UForm.
  • Entender como una tabla recibe datos y columnas declarativas.
  • Practicar operaciones de arrays: filter, map, some y find.
  • Traducir requerimientos de producto a criterios de aceptacion verificables.

5. Modelo de datos

El modelo representa la informacion minima necesaria para administrar clientes en una interfaz comercial. Esta definido en TypeScript con una interfaz llamada Cliente y un tipo union llamado ClienteEstado. 

type ClienteEstado = 'activo' | 'inactivo' | 'pendiente'

interface Cliente {
  id: number
  nombre: string
  email: string
  telefono: string
  ciudad: string
  estado: ClienteEstado
  fechaRegistro: string
}
Campo Tipo Uso en la interfaz Regla
id number Identificador visible en la tabla. Se genera automaticamente al crear.
nombre string Nombre principal del cliente. Minimo 2 caracteres.
email string Se muestra debajo del nombre y se puede copiar. Formato email y no duplicado.
telefono string Contacto telefonico. Minimo 7 caracteres.
ciudad string Ubicacion del cliente. Minimo 2 caracteres.
estado activo | inactivo | pendiente Badge visual y filtro. Debe pertenecer a la lista permitida.
fechaRegistro string Fecha formateada en la tabla. Se asigna automaticamente al crear.

6. Funcionalidades principales

Resumen superior

Muestra cuatro indicadores calculados con computed: total, activos, pendientes e inactivos. Cada tarjeta reacciona automaticamente cuando cambia la lista.

Busqueda

Permite buscar por nombre, correo, telefono o ciudad. La busqueda no modifica la data original; solo genera una version filtrada.

Filtro por estado

El usuario puede ver todos los clientes o limitar la lista a activos, inactivos o pendientes.

Tabla paginada

La tabla usa UTable y getPaginationRowModel para paginar de 8 en 8 registros.

Seleccion multiple

Cada fila tiene un checkbox. Al seleccionar clientes aparece el boton de eliminar seleccionados.

Menu de acciones

Cada fila contiene opciones para editar, copiar correo o eliminar.

Logica de filtrado

La lista que se renderiza en la tabla no es directamente clientes, sino clientesFiltrados. Esto mantiene separada la fuente de datos original de la representacion filtrada. 

const clientesFiltrados = computed(() => {
  const term = search.value.trim().toLowerCase()

  return clientes.value.filter((cliente) => {
    const matchesSearch = !term || [
      cliente.nombre,
      cliente.email,
      cliente.telefono,
      cliente.ciudad
    ].some(value => value.toLowerCase().includes(term))

    const matchesEstado = estadoFiltro.value === 'todos'
      || cliente.estado === estadoFiltro.value

    return matchesSearch && matchesEstado
  })
})

7. Arquitectura frontend

El CRUD vive principalmente en un solo componente de pagina: src/pages/customers.vue. Para fines pedagogicos, esto permite que los estudiantes vean todo el flujo en un mismo archivo: tipos, estado, validaciones, funciones y plantilla.

Capa Elemento Responsabilidad
Datos clientes Lista persistida de clientes mediante useStorage.
Estado UI formOpen, deleteOpen, rowSelection Controla modales, seleccion de filas y comportamiento visible.
Formulario state, schema, onSubmit Administra entradas del usuario y validacion.
Derivados clientesFiltrados, resumen, modalTitle Calcula informacion a partir del estado base.
Presentacion UDashboardPanel, UTable, UModal Renderiza la interfaz usando componentes de Nuxt UI.

Persistencia local

const clientes = useStorage<Cliente[]>('clientes-crud-v1', clientesIniciales)

useStorage conecta una variable reactiva con el almacenamiento del navegador. Cuando clientes.value cambia, tambien cambia el contenido guardado en localStorage. Si se recarga la pagina, la lista se restaura desde el navegador.

Limitacion importante: esta persistencia es local por navegador. Si otro usuario abre la aplicacion en otra computadora, no vera los mismos datos. Para compartir informacion entre usuarios se necesita backend.

8. Flujos del CRUD

Crear cliente

El usuario presiona el boton Nuevo cliente.
La funcion openCreateModal() limpia el formulario, asigna editingId = null y abre el modal.
El usuario completa nombre, correo, telefono, ciudad y estado.
Al enviar, UForm valida con Zod.
Si el email no esta duplicado, se calcula un nuevo ID y se agrega el cliente al inicio del array.
El modal se cierra, se limpia el formulario y aparece una notificacion de exito.
const nextId = Math.max(1000, ...clientes.value.map(item => item.id)) + 1

clientes.value = [{
  id: nextId,
  ...cliente,
  fechaRegistro: new Date().toISOString().slice(0, 10)
}, ...clientes.value]

Leer clientes

La tabla recibe :data="clientesFiltrados".
Las columnas se definen en el arreglo columns.
El usuario puede ordenar por cliente, paginar, ocultar columnas y seleccionar filas.
Si no hay resultados, se muestra una alerta con el mensaje No se encontraron clientes.

Actualizar cliente

El usuario abre el menu de acciones de una fila y elige Editar.
openEditModal(cliente) guarda el ID en editingId y copia los datos al formulario.
El modal cambia su titulo a Editar cliente gracias a modalTitle.
Al enviar, onSubmit detecta que existe editingId y ejecuta un map.
Solo el cliente cuyo ID coincide se reemplaza con los datos nuevos.
clientes.value = clientes.value.map(item => item.id === editingId.value
  ? { ...item, ...cliente }
  : item)

Eliminar cliente

Para eliminar uno, el usuario selecciona Eliminar desde el menu de acciones.
Para eliminar varios, el usuario marca checkboxes y presiona Eliminar.
El sistema guarda los IDs a eliminar en deleteIds.
Se abre un modal de confirmacion para evitar eliminaciones accidentales.
Al confirmar, se usa filter para dejar solo los clientes cuyo ID no esta en el conjunto de eliminacion.
const ids = new Set(deleteIds.value)
clientes.value = clientes.value.filter(cliente => !ids.has(cliente.id))

9. Validaciones y reglas de negocio

La validacion tiene dos niveles: validacion de forma con Zod y validacion de negocio para evitar emails duplicados. 

const schema = z.object({
  nombre: z.string().min(2, 'Ingresa al menos 2 caracteres'),
  email: z.string().email('Ingresa un correo valido'),
  telefono: z.string().min(7, 'Ingresa un telefono valido'),
  ciudad: z.string().min(2, 'Ingresa la ciudad'),
  estado: z.enum(['activo', 'inactivo', 'pendiente'])
})
Regla Motivo Respuesta del sistema
Nombre minimo de 2 caracteres Evita registros vacios o poco descriptivos. Muestra error en el formulario.
Email con formato valido Permite usar el correo como contacto confiable. Muestra error en el formulario.
Email unico Evita duplicar el mismo cliente. Muestra toast: Correo duplicado.
Telefono minimo de 7 caracteres Evita telefonos demasiado cortos. Muestra error en el formulario.
Estado dentro de lista permitida Garantiza consistencia visual y de filtros. Solo se puede elegir activo, inactivo o pendiente.

11. Requerimientos funcionales y no funcionales

Requerimientos funcionales

ID Requerimiento Prioridad
RF-01 El sistema debe listar clientes registrados en una tabla. Must
RF-02 El sistema debe permitir crear clientes con datos validados. Must
RF-03 El sistema debe permitir editar clientes existentes. Must
RF-04 El sistema debe permitir eliminar uno o varios clientes. Must
RF-05 El sistema debe impedir emails duplicados. Should

RF-07 El sistema debe persistir cambios en el navegador. Should
RF-08 El sistema podria permitir copiar el correo desde una fila. Could

Requerimientos no funcionales

  • Usabilidad: la pantalla debe ser clara para usuarios principiantes.
  • Responsive: los controles deben funcionar en escritorio y pantallas pequenas.
  • Mantenibilidad: la logica debe estar organizada en funciones con nombres descriptivos.
  • Performance: el filtrado debe ser suficiente para listas pequenas o medianas en memoria.
  • Accesibilidad basica: los botones, inputs y modales deben usar componentes accesibles de Nuxt UI.

12. Criterios de aceptacion

Escenario Dado Cuando Entonces
Crear cliente valido El usuario abre el modal de nuevo cliente. Completa todos los campos correctamente y envia. El cliente aparece en la tabla y se muestra un toast de exito.
Crear cliente invalido El formulario esta abierto. El usuario deja campos incompletos o email invalido. El formulario muestra errores y no crea el registro.
Email duplicado Ya existe un cliente con un correo. Se intenta crear o editar otro cliente con el mismo correo. El sistema muestra Correo duplicado y cancela el guardado.
Editar cliente Existe un cliente en la tabla. El usuario elige editar, modifica datos y guarda. La fila se actualiza sin duplicar el registro.
Eliminar cliente Existe un cliente en la tabla. El usuario confirma su eliminacion. El cliente desaparece de la tabla y de localStorage.
Filtrar por estado Hay clientes con distintos estados. El usuario selecciona un estado en el filtro. La tabla muestra solo clientes del estado elegido.
Persistencia El usuario crea, edita o elimina clientes. Recarga el navegador. Los cambios se mantienen porque fueron guardados localmente.

13. Material sugerido para clase

Secuencia de explicacion

  1. Mostrar la interfaz final y preguntar que operaciones CRUD detectan los estudiantes.
  2. Explicar el modelo Cliente y por que TypeScript ayuda a evitar errores.
  3. Revisar el estado base: clientes, search, estadoFiltro, pagination.
  4. Explicar computed con el ejemplo de clientesFiltrados y resumen.
  5. Analizar el formulario y sus validaciones con Zod.
  6. Recorrer cada flujo: crear, leer, actualizar y eliminar.
  7. Mostrar como localStorage conserva cambios al recargar.
  8. Proponer mejoras para practicar.

Preguntas para estudiantes

  • Por que clientesFiltrados debe ser un computed y no una variable normal?
  • Que pasaria si editaramos directamente el arreglo original sin usar map?
  • Por que conviene confirmar antes de eliminar?
  • Que diferencia hay entre validacion de formulario y regla de negocio?
  • Que limitaciones tiene localStorage frente a una base de datos real?

Resumen final

Este CRUD demuestra que una aplicacion frontend puede cubrir una gran parte del ciclo de gestion de datos: capturar informacion, validarla, mostrarla, actualizarla, eliminarla y persistirla localmente. La siguiente evolucion natural es conectar estas mismas operaciones a una API real, donde clientes.value dejaria de ser la fuente definitiva y pasaria a sincronizarse con un servidor.

Documento generado como material de apoyo para estudiantes de desarrollo web frontend.
